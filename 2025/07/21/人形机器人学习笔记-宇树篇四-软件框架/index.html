1. 系统构成

本工程是一个基于ROS的机器人项目，集成了运动控制、传感器驱动、仿真、导航等多个功能模块。

各个功能包之间通过ROS消息进行通信，便于模块化开发和扩展。

同时，工程中包含了丰富的第三方库和示例代码，方便开发者学习和使用。

  • devel/ - ros编译后生成的可执行文件存放区，包含头文件、库文件、环境脚本配置文件

  • build/ - ros编译过程中产生的中间文件

  • src/ - ROS工作空间的源码目录，包含了各个功能包（package），如unitree_guide、unitree_ros等

    • example/ - 示例代码区域，演示如何调用机器人的运动接口

    • unitree_guide/ - 机器人引导、导航相关的功能包，包含源码、配置、数据

      • data/ - 存放数据文件、配置、地图等

        • cofig/ - 通常用于存放配置文件，比如参数表、机器人模型参数、传感器参数、路径规划参数等。

          这些配置文件会在程序启动或运行过程中被加载，用于调整系统的行为和性能。

          • rviz.rviz - RViz (ROS的一个可视化工具)的可视化配置文件，里面详细保存了 RViz 各种显示项、界面布局和参数设置，可以每次设置后保存rivz文件，后续直接打开详情细节看后续。

        • policy/- 一般用于存放策略相关的数据文件，比如运动控制策略、决策逻辑、行为树、状态机描述等。当前文件夹支持了机器人运动控制的“策略推理”部分，通常用于实现智能化、自主化的运动决策。

          • actor_1.mnn - MNN格式的运动策略神经网络模型，供机器人实时推理使用

          • actor_1.onnx - ONNX格式的原始策略神经网络模型，便于跨平台迁移和转换，ONNX 是一种通用的神经网络模型交换格式，便于在不同深度学习框架之间迁移模型。该文件一般是训练后导出的原始策略网络，可以用于推理或转换为其他格式（如 MNN）。

          • g1.yaml - 运动策略相关的参数配置文件，辅助模型推理和应用

      • include/ - 存放头文件，C++接口声明

      • launch/ - ROS启动文件，定义节点启动和参数配置

        • gazebo.launch - 用于在 Gazebo 仿真环境中启动 Unitree 机器人模型，并加载相关控制器和参数
        • rosbag.launch - 用于回放 rosbag 录制的数据包，并同步启动 RViz 可视化和机器人状态发布。

      • src/ - 源代码目录，主要功能实现

        • control/ - 实现具体的控制算法，将决策转化为底层动作

        • FSM/ - 实现有限状态机，管理机器人各类行为状态及切换

          • FSM.cpp - 有限状态机主类实现，负责状态调度与切换

          • FSMState.cpp - 状态基类实现，提供统一接口，初始化 FSMState 状态对象的基本信息（类型、名称），绑定底层控制命令和状态数据的指针，方便在状态逻辑中直接访问和操作机器人底层硬件，在每次new一个状态对象时（FSM.cpp），都会调用对应的构造函数

          • State_FixedStand.cpp - 固定站立状态实现，控制机器人站立（下方有注释）

          • State_locomotion.cpp - 运动状态实现，加载并运行策略网络，实现自主运动（下方有注释）

          • State_Passive.cpp - 被动状态实现，控制机器人进入安全/待机模式

        • interface/ - 实现与外部系统（硬件、传感器、ROS等）的接口与通信

        • main.cpp: 程序主入口，负责初始化、模块集成和主循环

      • thirdParty/ - 第三方库或外部依赖代码

        • MNN - 机器人系统提供高效、轻量级的神经网络模型推理能力，使得深度学习模型可以在机器人本地实时运行，支撑如运动控制、感知等智能功能。
        • yaml-cpp - 为 C++ 代码提供 YAML 配置文件的解析和读取能力，方便开发者将参数、配置等信息以 YAML 文件形式管理，提高系统的灵活性和可维护性。如: data/policy/g1.yaml

      • CMarkList.txt - 构建脚本，定义编译规则和依赖

      • package.xml - ROS包元信息，声明依赖和基本信息

    • unitree_ros/ - 集合包，里面包含了多个与Unitree机器人相关的功能包

    • unitree_ros_to_real/ - 仿真到真实机器人切换的适配包，包含消息定义（unitree_legged_msgs）

      CMarkList.txt - ROS工程自动创建，用来组织软件系统编译结构。

2. 代码逻辑

Step1：开始实时操作系统；设置输出格式

Step2：初始化一个名为**nitree_gazebo_servo**的节点，用于伺服控制

Step3：初始化接口对象，保存UserValue（前后、左右、旋转的速度参数），选择机器人仿真平台；

Step4： 开启ROS的关于伺服电机的订阅者和发布者节点，捕获Ctrl+C关闭节点信号，创建键盘输入对象；

  • KeyBoad();:

    • 初始化用户命令，设置用户值为0；-> 设置终端显示方式 -> 开始运行键盘（runKeyBoard），并向创建线程进行监听，使其可以实时监听键盘输入。

    • runKeyBoard() -> run() : 判断是否有键盘输入，若有则先检查检查命令：是否输入为1-8/Space,；

      键盘键值分别代表为：

      1 UserCommand::L2_B;
      2 UserCommand::L2_A;
      3 UserCommand::L2_X;
      4 UserCommand::START;
      0 UserCommand::L1_X;
      9 UserCommand::L1_A;
      8 UserCommand::L1_Y;
      Space userValue.setZero(); UserCommand::NONE;
      default UserCommand::NONE

      同时判断是否输入W\S\A\D，若输入则增加对应的uservalue值；

Step5： 创建一个控制元件对象new CtrlComponents(ioInter);（在CtrlComponents.h中），实现底层控制器与接口的连接，并设置控制元件参数；

Step6：创建控制分层ControlFrame ctrlFrame(ctrlComp);，里面运动FSM->run()函数，此时启动有限状态机；

  • 在FSM->run()函数中会将，机器人运动模式设置为，_stateList.passive;被动模式，会一直运行 _nextStateName = _currentState->checkChange();函数（State_Passive.cpp中），同时，检查机器人状态是否改变；
    • 若按下键盘2，即UserCommand::L2_A;则机器人状态则会到FSMStateName::FIXEDSTAND;模式下。此时，便会运行State_locomotion->run()；中内容，此时机器人会将腿部电机初始化到目标位置；同时，一直检查机器人是否切换状态。此时，L2_B（1键）为返回PASSIVE状态；START（4键）为进入LOCOMOTION状态；（实现逻辑在State_FIxedStand.cpp/FSMStateName State_FixedStand::checkChange()）中。

Step7：若按下4，则执行locomotion的run的内容，run具体实现内容在后续中。

3. 关键代码详解

3.1 unitree_guide/data/policy/g1.yaml

1. 与模型结构的关系

  • num_obs: 47

表示策略神经网络的输入层有47个神经元，对应47维观测量。

  • num_actions: 12

表示策略神经网络的输出层有12个神经元，对应12个动作（通常是12个腿部关节的目标位置/速度/力矩等）。

  • 归一化参数（如 action_scale, cmd_scale 等）

用于对输入观测和输出动作进行缩放，使其分布适合神经网络推理，提升模型的泛化能力和稳定性。

  • policy_path

指定了实际加载的策略模型文件路径，模型结构（如隐藏层、激活函数等）由该文件决定，yaml文件只负责参数配置。

2.为什么不用 policy 目录下的 actor_1.onnx 策略？

  1. 用途不同

    • motion.pt 可能是训练时用的 PyTorch 原生模型，适合在 Python 环境下直接加载和推理（如 legged_gym 训练/测试脚本）。

    • actor_1.onnx 是经过导出的通用格式，适合跨平台部署（如C++、嵌入式、推理引擎等），但不一定直接用于训练环境。

  2. 加载方式不同

    • 如果代码是基于 PyTorch 的，直接加载 .pt 文件最方便。

    • 如果代码是基于 ONNX Runtime、MNN 或其他推理引擎，则会加载 .onnx 或 .mnn 文件。

  3. 配置文件的适配性

    • g1.yaml 可能是直接从训练环境（如 legged_gym）导出的，默认指向训练用的 .pt 文件。

    • 实际部署到机器人时，通常会用 ONNX 或 MNN 格式的模型，并在代码或配置中指定 actor_1.onnx 或 actor_1.mnn。

  4. 模型版本和一致性

    • motion.pt、actor_1.onnx、actor_1.mnn 这几个文件理论上是同一个模型的不同格式，但实际部署时要确保它们是同一版本、同一训练结果导出的。

3. 实际使用建议

  • 训练/仿真阶段：一般用 .pt（PyTorch）模型，方便调试和训练。

  • 部署/推理阶段：一般用 .onnx 或 .mnn 等跨平台格式，适合在机器人或嵌入式设备上高效推理。

  • 如果你要用 policy 目录下的 actor_1.onnx，需要在配置文件中把 policy_path 改成 actor_1.onnx 的实际路径，并确保你的加载代码支持 ONNX 格式。

3.2 unitree_guide/launch/gazobo.launch

Step1：首先，设置仿真参数（如世界文件、机器人型号、是否暂停、是否显示GUI等）。

Step2：利用仿真参数，启动 Gazebo 空世界（empty_world.launch），加载指定的世界文件。

Step3：用 xacro 工具生成机器人 URDF，并加载到 ROS 参数服务器（robot_description）。

​ 注：xcro工具用于ROS的XML宏语言，通过常量和宏定义特性简化URDF文件的编写

Step4：通过 gazebo_ros/spawn_model 节点，将机器人模型实例化到仿真环境中。

Step5：加载关节控制器参数（robot_control.yaml）。

Step6：启动 controller_manager/spawner，加载并启动各个关节控制器。

Step7：启动 robot_state_publisher，将关节状态发布为 TF，便于 RViz 等工具可视化。

Step8：加载 unitree_controller 的控制参数。

1. /robot_description 的作用

(1) 存储机器人的 URDF/Xacro 模型

  • 通常通过 <param> 在 launch 文件 中加载：

或直接加载 URDF：

  • <param name="robot_description" 
       textfile="$(find my_robot)/urdf/robot.urdf" />

2. TF（Transform）

  • 作用：存储和广播不同连杆之间的 坐标系变换关系（平移 + 旋转）。

3.3 src/文件夹

3.3.1. /control/ControlFrame.cpp

  • ControlFrame 是一个控制框架类，主要负责集成和调度 FSM（有限状态机）控制器。

  • 它通过 run() 方法驱动 FSM 控制器的运行，实现对机器人行为的高层管理。

  • 这种设计有利于将控制流程与具体状态逻辑解耦，结构清晰、易于扩展。

3.3.2 /FSM/State_FixStand.cpp

1. void State_FixedStand::run()

// 平滑插值：从起始位置到目标位置

_duration:在State_FixedStand.h中，每次 _percent加1 / _duration，让腿部慢慢到期望目标点。

_target:在State_FixedStand.h中,腿部各个关节的期望位置。

_percent = _percent > 1 ? 1 : _percent; 三元判断语句： _percent>1为真时， _percent = 1,反之为 _percent

3.3.3 State_locomotion.cpp

1. void State_locomotion::loadPolicy()

  • 作用:加载训练策略;
  • 步骤:
  • 根据模型文件创建解释器Interpreter;
  • 根据ScheduleCOnfig配置创建配置会话Session
  • 设置输入数据归一化因子,和平均值,为后续运行推理网络作准备; 为什么要将输入数据归一化?

2. void State_locomotion::inference()

  • 作用:运行推理模型,同时记录执行次数
  • 步骤:
  • 获取机器人实际数据:IMU数据3/重心所处位置3/用户期望(前后.左右.旋转速度)/腿部电机的位置和速度信息24/前一次网络输出的电机位置信息12/机器人相位信息2 = 47
    • 相位信息的作用：
      • 功能：生成机器人的步态时钟信号，实现周期性运动控制。
      • 工程意义：使神经网络能感知时间维度，输出协调、平滑的动作。
      • 扩展性：可通过调整 period 或叠加多个频率信号实现更复杂的步态（如跳跃、上下楼梯）。
  • 输入信息归一化处理；
  • 将处理有后的数据拷贝到神经网络输入张量中；
  • 运行推理模型；
  • 获得网络输出值；
  • 并将其进行反归一化处理；

3. void State_locomotion::run()

  • 将神经网络得到的输出，通过底层控制给电机；
  • 神经网络的输出为12个腿部电机关节的位置信息；
